# 概念设计引擎 — 全局规则

> 本文件是概念设计引擎的**唯一规则来源**，所有项目均受其约束。
> 项目特定规则放在 `projects/<项目名>/`。

---

## Layer 1：类型学诊断树（诊断器）

在生成任何几何体之前，AI **必须**输出结构化的类型学诊断报告：

```
=== 类型学诊断报告 ===
建筑类型：         [酒店/公寓/度假村/办公/综合体/文化/混合...]
定位级别：         [经济型/中档/中高档/奢华]
总建筑面积：       [XX,XXX ㎡]
场地特征：         [城市核心区/郊区/山地/滨海/...]
核心设计矛盾：     [至少列出 2 条，如：垂直声学冲突 /
                    采光与进深最大化的矛盾 /
                    景观面最大化与红线约束的冲突]
功能组团划分：     [将楼层归并为 3-4 个巨型组团，如：
                    裙房 1-4F / 办公 5-15F / 客房 16-26F]
推荐适用母题：     [列 2-3 个，如：Monolith / Terrace / Interlock]
========================
```

### 建筑类型与行业公理

| 类型 | 行业公理 | 核心矛盾锚点 |
|:----|:---------|:------------|
| **商务酒店** | 下公上私，下闹上静 | 垂直声学阻尼；四流线分流；核心筒效率 |
| **高端公寓** | 面宽最大化，户户朝南 | 日照阴影线控制；开间进深比（≥1:1.5）；标准层重复率 |
| **野奢度假村** | 景观最大化，分散式布局 | 地形等高线单向展开；视线扇面；低密度零扰 |
| **总部办公** | 高效标准层，灵活分隔 | 核心筒偏置或外挂；标准层进深 ≤18m；立面统一模数 |
| **商业综合体** | 多业态水平串联 | 人流动线组织；中庭空间序列；后勤与客流完全隔离 |
| **文化/展览** | 空间叙事性 | 大跨度结构；流线引导；自然采光 |
| **混合功能** | 综合上述策略 | 优先级：流线 → 声学 → 结构 |

### 功能组团划分规则

1. **同一功能的连续楼层必须归为一个组团**
2. **功能转换楼层 = "转换节点"**——允许在此处发生退台/悬挑/结构转换
3. **每个组团内部的外墙线默认 100% 垂直平齐**
4. **组团数量：2-4 个**（不含地下和屋顶设备层）

## Layer 2：母题语法库

每个方案必须严格绑定**一种**母题，禁止在一个方案内混用。

### 母题 A：Monolith（极简巨构）

| 字段 | 内容 |
|:----|:------|
| **参考案例** | 密斯·凡·德·罗 - 西格拉姆大厦；SOM - 汉考克中心；安藤忠雄 - 光的空间 |
| **核心特征** | 外立面 100% 垂直平齐；整体性是最高准则 |
| **适用条件** | 城市核心区；限高严格；甲方要求"稳重高档"；≥20F |
| **体块逻辑** | 生成最大配平面积的刚性外壳 → 通过 BooleanDifference 挖中庭/入口/天井减面积。外皮在任何楼层都不允许出现凹凸进退 |
| **面积消纳策略** | 差值全部转化为内部通高中庭、入口凹洞、天井。**严禁在外立面产生锯齿** |
| **顶部策略** | 塔楼与裙房外皮完全对齐。顶部允许 ≤3m 的阴角刻槽收头。不鼓励悬挑 |

### 母题 B：Terrace（阶梯错叠）

| 字段 | 内容 |
|:----|:------|
| **参考案例** | BIG - 8 House / Via 57 West；藤本壮介 - 东京公寓；MVRDV - 书山 |
| **核心特征** | 多个组团间整体跌落；退台面转化为露台/绿化/公共空间；单向退缩 |
| **适用条件** | 沿主干道；有景观面要求；需要公共空间补偿；中高层建筑 |
| **体块逻辑** | 锁定主立面边界（通常为东/南），所有面积差沿次要方向单向退缩。退让幅度 ≥3m/次，退台面标记为 `Terrace_Zone` |
| **面积消纳策略** | 每级退台暴露下方组团顶部 → 转化为露台。退台方向必须一致，禁止东西交错 |
| **顶部策略** | 塔楼保持与裙房相同退缩方向。允许退台面做 2-3m 悬挑观景平台 |

### 母题 C：Interlock（异质穿插）

| 字段 | 内容 |
|:----|:------|
| **参考案例** | OMA - 央视总部；扎哈 - 北京大兴机场；蓝天组 - 宝马世界 |
| **核心特征** | 裙房与塔楼作为独立几何体互相咬合；悬挑作为结构叙事；强烈的视觉张力 |
| **适用条件** | 地标性项目；转角地块；甲方要求"先锋感"；不限高或不严格 |
| **体块逻辑** | 裙房和塔楼各自建立独立几何中心轴。两者在 Z 轴上允许物理交叉咬合。面积微调通过调整咬合深度逼近 |
| **面积消纳策略** | 塔楼从裙房间"穿出"，咬合区重叠面积计入裙房或塔楼。悬挑产生的面积增量由对侧配重/收缩抵消 |
| **顶部策略** | 顶部楼层允许大尺度悬挑（≤6m）作为天际线收头。方向单一，避免结构扭转 |

### 母题选择规则

```
IF 酒店 AND 定位 ≥ 中高档 AND 限高严格：
    推荐 = [Monolith, Terrace, Interlock]

IF 公寓 AND 面宽要求高：
    推荐 = [Terrace, Monolith]
    排除 Interlock（面宽损失大）

IF 度假村 AND 低密度分散：
    推荐 = [Terrace]
    排除 Monolith（过于集中）和 Interlock（过于激进）

IF 办公 AND 标准化要求高：
    推荐 = [Monolith, Interlock]

IF 综合体 AND 地标性要求高：
    推荐 = [Interlock, Terrace]

IF 文化：
    推荐 = [Interlock, Monolith]
```

### 组团化体块生成规范

1. **严禁逐层堆叠**。每个组团生成一个完整体块，所有楼层共用同一外轮廓
2. **组团数量 2-4 个**，超出需说明理由
3. **每个体块必须指派功能标签**（如：裙房、办公、客房）
4. **核心筒作为独立的贯通体块**，在所有组团中保持完整贯通

## Layer 3：刚性合规器

### 3.1 面积配平校验

```
组团总面积 = Σ(每层目标面积)     ← 从任务书读取
体块总面积 = Σ(组团体块 BoundingBox 算出的面积)
误差率 = |体块面积 - 目标面积| / 目标面积
要求：≤ 5%
```

### 3.2 规范检查（按类型加载）

| 类型 | 适用规范 | 检查项 |
|:----|:---------|:-------|
| 酒店 | JGJ62-2014 / GB 51165 | 客房净面积 ≥25㎡；走廊 ≥1.5m；层高 ≥3.3m；无障碍客房 ≥2 间 |
| 公寓 | GB 50096 | 套型面积；日照时数；采光系数；面宽比 |
| 办公 | JGJ 67 | 进深 ≤18m；净高 ≥2.6m；防火分区面积 |
| 通用 | 各地规划条例 | 密度 ≤35%；容积率 ≤4.0；限高；退线；绿地率 |

### 3.3 红线校验

- 所有体块的 XY 坐标必须在退线范围内
- 超限时自动修正：优先缩减裙房边界，再缩减塔楼
- 三次修正后仍超限 → 切换母题

### 3.4 碎边消灭法案

- 相邻两层面积差 ≤150㎡ 或进退 ≤2m → **禁止**在外立面产生锯齿
- 差值通过内部核心筒偏心、走廊宽度调节、中庭扩缩消化
- 退台仅允许在组团分界处发生（≥3m）

### 3.5 超限回退协议

```
IF 误差率 > 5% OR 规范校验不通过：
    Step 1：调整组团内部核心筒位置（偏心 ≤2m）
    Step 2：调整组团之间的退台/悬挑幅度
    Step 3：如果前两步仍不满足 → 切换母题重做
```

## 后合理化说明

体块生成完成后，必须输出后合理化说明，解释每个体块动作对应的任务书条款：

```
=== 后合理化说明 ===
方案 [A/B/C] —— [母题名称]

[体块动作] → [任务书条款]
  [具体说明]

例：
3F 西侧 800㎡ 退台 → 4F 宴会厅 1,850㎡ 相比 3F 餐饮 2,650㎡ 减少 800㎡
  → 转化为 3F 餐饮层的室外观景露台

16F 四周内缩 → 客房标准层从办公 1,350㎡ 退台至 1,210㎡
  → 退台面形成 16F 麻将客房的专属空中花园
========================
```

## 附录（通用，不依赖项目类型）

### A. 单位铁律
所有坐标值使用**毫米（mm）**。1m = 1000mm。

### B. 图层命名规范
```
[方案字母 A/B/C]
├── 控制线
├── [功能组团 1]
│   ├── [功能细分]（按色码着色）
│   └── ...
├── [功能组团 2]
├── [功能组团 3]
└── 核心筒
```

图层名禁止使用 `::`（Rhino 会解释为嵌套分隔符，产生大量孤儿图层）。

### C. 色码规范

| 功能 | RGB | 颜色名 |
|:----|:----|:-------|
| 公共区（大堂/宴会厅） | (0,150,230) | 蓝 |
| 商业/会议室 | (0,100,180) | 深蓝 |
| 餐饮 | (232,168,56) | 橙 |
| 办公 | (74,217,163) | 绿 |
| KTV/SPA/娱乐 | (180,100,200) | 紫 |
| 客房（偶数层） | (232,168,56) | 橙 |
| 客房（奇数层） | (255,200,100) | 浅橙 |
| 后勤/设备 | (155,155,155) | 灰 |

### D. 标注规范
每个方案用 TextDot 标注方案名称，放在场地右上角外侧。

### E. 核心筒原则
核心筒（1F-顶F 贯通）必须在塔楼内部，不在边缘。

## 安全协议

### 协议 1：路径总督
- AI 的读写操作**限制**在当前 `projects/<项目名>/` 目录
- 未经明确授权，禁止系统级文件访问或跨项目污染
- 项目配置必须从 `projects/<项目名>/config.json` 加载

### 协议 2：防死循环
- 如果生成尝试失败（拓扑冲突、面积误差 >5%、布尔运算失败）：
  - **第一次失败**：记录错误，调整参数重试一次
  - **第二次连续失败**：**立即停止**，输出详细错误分析，禁止盲目重试
  - **强制暂停**：等待用户指令后再进行下一步

### 协议 3：图层卫生
- 图层名禁止使用 `::`——Rhino 会将其解释为嵌套分隔符，产生大量孤儿图层
- 使用 `ParentLayer()` 建立父子层级
- 每轮迭代（R1、R2...）有独立的图层根节点
